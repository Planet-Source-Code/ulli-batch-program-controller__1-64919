Description

cPS is a class which will govern batch-type programs having one or more sequential input data streams (normally files) and a number of control-breaks levels. To do this cPS raises action request events to direct the client program to take the correct actions (programmed into action modules) for each situation. The client program in turn performs the requested action, informs cPS of the result (if any) and then waits for the next action request. With an AMD 1800 MHz Athlon CPU it takes roughly 6 µsecs average until cPS has determined what the client program should do next and consequently fires the next action event. cPS supports a variety of properties to let the client program configure a run, to enable the client program to inform cPS of various situations or to inform the client of certain aspects of the current run.

The client program is structured into a number of modules, each being responsible for the execution of one action or subtask of the complete task. These modules are grouped into action types (like for example Fetch Next Record  or Close Previous Group) and most action types are further subdivided into action numbers for each data stream (file) or each group-level (control break) respectively.

To configure and then start a run the client program supplies the values for the appropriate properties and then calls the Run method. cPS will check the configured properties to be consistent among themselves and will then fire events informing the client program of errors (if the properties are inconsistent) or to direct the client program to the next action module to execute

Finally the Run method returns and the run is completed.

The client program types include: Report Writers with subtotals and/or totals, Grouped or Categorized Statistics, Batch Updates that update a master file from one or more transaction files, and combinations thereof. As cPS is not aware of the data sources and only requires the keys of the current records, data stream sources can be anything from flat sequential files via database tables, views or joins  to sorted or unsorted in-memory arrays. Therefore in this document the word 'stream' rather than 'file' is used to describe a succession of records. It is the client program's responsibility to fetch or get the next record from a data stream when directed to do so by cPSs action request events. cPS will then query the client program for each records keys (or EOF in case there are no more records in a stream) and decide on the next action the client program will have to take.

cPS incorporates an internal merging logic which is activated when more than one data stream is controlled by cPS. Obviously for this merging to work each data stream must be ordered by identical keys. This feature assures that corresponding (by keys) records from different data streams are present in memory before the record processing modules are invoked. Thus it is possible to exchange or combine data between trans­action records and their corresponding master record. The mutual relationship between the records present in memory can be found in the RecordIs property or in the RecRelation property. When keys from different data streams are equal, the merging logic uses the data stream number (derived from the ActionNumber within ActionType 3 (PS_CheckRecord) that supplied the keys) to determine the processing sequence of these.

When the keys selected for processing (by the merging logic, if more than one stream is controlled) differ from the previous keys, cPS will invoke a control break series consisting of group footers for the previous keys and then the group headers for the next keys. Group headers are also invoked when there are no previous keys (start of run) and group footers are invoked after EOR (end of run) detection. For each key there is one control break level consisting of group headers and group footers, and a control break in a higher level will automatically trigger the subordinate levels. ActionNumber 0  of ActionType 5 or 7 resp. will be requested once during a run and may be used to initialize and process grand totals for example.

There is one specialty to watch out for however. When the group footers are invoked the record areas in memory already hold the records belonging to the next key group. cPS cannot detect a control break until it has knowledge of the succeeding keys. This in turn requires that the next records from each stream are in memory so that the client program can extract their keys and communicate those to cPS via the RecordKey properties.

When all data streams are exhausted, cPS will then direct the client program to the group footers (to close the last key groups) and thence to the termination module.

cPS supports a Trace Option with several TraceRequest levels which uses the Trace event to communicate with the client program.

 

Before you can use a cPS class in your application, you must add the WINPS.DLL file to your project. If you use the object in most of your VB projects, you may want to add it to VB's Autoload file.

To distribute applications you create with the cPS class, you must install and register it on the user's computer. The Setup Wizard provided with Visual Basic provides tools to help you do that. Please refer to the Visual Basic manual for details.

GroupIsVoid Property

Description          Returns whether the current group level processing could be omitted.

Syntax                   val = cPS.GroupIsVoid

Accessible in       PS_CloseGroup

Remarks          This property applies to the current group footer level only. It returns True when it can safely be assumed that the current accumulated subtotal will be repeated in the next group footer level. An example is best used to clarify the situation.

Assuming an invoicing program with two control break levels (date and order number). If there is one order only for a given date then the invoice would basically look like this:                         

AT/AN 5/2                            Date  8.7.2003

AT/AN 5/1                                Order Number 4711

AT/AN 6/1                                    Item Number 0815              3.95

AT/AN 6/1                                    Item Number 0816              4.95

AT/AN 7/1                                Subtotal for Order 4711         8.90*

AT/AN 7/2                            Total for Date 8.7.2003             8.90**

However, the line AT/AN 71 is essentially repeated in line AT/AN 72 and could therefore be suppressed. GroupIsVoid returns True when this situation arises  (in this particular case in AT/AN 71) and the invoice could be modified to the following form: 

AT/AN 5/2                            Date  8.7.2003

AT/AN 5/1                                Order Number 4711

AT/AN 6/1                                    Item Number 0815              3.95

AT/AN 6/1                                    Item Number 0816              4.95

AT/AN 7/2                            Total for Date 8.7.2003             8.90**

                        Note: AT/AN stands for ActionType/ActionNumber.

Data Type             Boolean
